Design system reference

The reconciled direction: an editorial scrapbook. Texture, hard shadows and rotation live only on brand surfaces — the header, tags, illustrations and empty states. Reading and input surfaces are flat, high-contrast and quiet. Companion to the prototype, breakpoint boards and the audit doc.

Colour roles

Palette doc is the source of truth. Blue is dominant; yellow is reserved for tags and the wordmark; orange for small accents only. All text pairs meet 4.5:1.

Typography

Exactly two families. Young Serif (variable, self-hosted) — Bold for the wordmark, Regular for all display headings and titles; Instrument Sans for everything else. Black Han Sans and Average are retired — the wordmark is Young Serif Bold, yellow on the orange header, with the hourglass icon in the lockup and a −0.4° rotation. Negative letter-spacing on serif display sizes only. Max line length ~70ch.

Surfaces, radii & shadows

Brand card

Light-blue construction paper (rgb(205,225,238)) with the cardboard texture multiplied in; 3px radius; hard shadow 0 4px 4px rgba(0,0,0,0.25). Recipe cards and title strips. Photos get the scanned-print overlay directly. The login/reset card is the same recipe-card treatment in brand yellow (rgba(239,218,107,0.78)).

Reading / input surface

Flat white, no texture, no shadow. Ingredients, method, every form field group.

Brand bar

Brand orange (rgb(216,74,26)) with paper texture multiplied in. Global header only.

Note post-it

Notes are thin post-its stuck to the page: cardboard texture over paper colour, square corners, panel shadow 1px 4px 2px rgba(0,0,0,0.25), ±0.4° rotation. Colours alternate yellow → blue → orange (rgba 0.78 washes of rgb(239,218,107) / rgb(205,225,238) / rgb(240,178,148)) in list order.

Tag chip

vegetarian

Cardboard-textured yellow, 4px radius, 1px hard shadow. The one “scrapbook” element allowed inside content.

Radii: 12px cards/dialogs · 8px inputs/buttons · 4px chips. Interactive height ≥44px. Focus: 2px #1F6590 outline, 2px offset, global.

Buttons

Brand action (cardboard-textured yellow, hard shadow) is reserved for the header’s “Add recipe”, Save and Edit recipe — at most one per view. Everything else uses flat primary/secondary. All paper buttons and chips share the physical press model: hover lifts (−1px translate, deeper offset shadow), press flattens (+1px translate, shadow removed).

Tags at scale

Tags are user-generated and unbounded, so tag pickers never render the full list as a checkbox grid. Two patterns:

Recipe form — combobox

pasta× + quick 3 + Add “fermented”

Selected tags are yellow paper chips with ×. One search field filters suggestions (12 most-used, with counts); an unmatched query offers a dashed “Add” chip; Enter creates.

Filter panel & sheet — searchable list

A “Find a tag” input above the scrolling checkbox list. Selected tags pin to the top; the rest sort by recipe count, then alphabetically. Empty results show “No tags match”. Active filters echo below the toolbar as removable yellow tag chips.

Motion & interactivity

Cards and chips behave like physical paper. All motion respects prefers-reduced-motion.

Card hover (desktop)Lifts −6px, tilts −0.5°, scales 1.01, shadow deepens to 0 14px 16px; photo straightens and scales 1.02; the recipe title underlines (3px offset) as a link affordance. 180ms ease.
Card press (mobile)Subtle press-down: scale 0.995, shadow flattens while touched.
Deck shuffle (desktop)On filter/search/sort change, cards collapse onto the top-left pile (alternating ±2.5° tilts) then deal back out, 460ms, 40ms stagger, FLIP via WAAPI.
Feed fade (mobile)Scroll-driven opacity fade-in only (animation-timeline: view(), entry 0–35%); no transforms, so press states never fight it.
Open recipeWhole card is the tap target (chips and star stop propagation). Clicked card lifts off the pile 170ms, then the page slides up and settles 280ms.
Tag chip tapAdds the tag to active filters; on a recipe page it returns to the filtered library; on mobile it smooth-scrolls to the top of the list.
Favourite starMuted star on a white round chip → pops onto a yellow paper chip with a 0.4s spin-in (scale 0.6→1.25→1, −20°→8°→0°).
Header linksWhite header text buttons (“‹ Back”) underline on hover (3px offset) like links. Logo click on the home page smooth-scrolls to top; from any other view it returns home.
Log in from a recipeLogged-out recipe pages show “Log in” in the header; signing in returns to the same recipe, never the library.
Save confirmationSave flips to a green “✓ Saved” with a 0.45s pop for 700ms before returning to the recipe and firing the success toast.

Toasts: success on green (#EAF3E6 / #4E9A5C / #1C4A26) for saved/created/favourite/sign-in; danger on red (#FFEDEF / #EBA9AF / #8C1D29) for deletions; neutral ink for everything else. 2.6s, bottom-centre.

Access control — what must sit behind auth

The prototype hides these from public users with client-side state. In the deployed product every one of them must be enforced server-side — hiding a control is not authorisation.

Recipe create / edit / delete“+ Add recipe”, the add/edit form, Save, “Edit recipe”, “Delete recipe”. Editor role required.
Recipe photosReplace / reframe / drag-drop upload on the recipe page. Cards are always view-only. Public users get read-only images.
NotesComposer, edit and delete (delete additionally scoped to the note’s author). Reading notes is public.
FavouritesStar buttons on cards and recipe pages, plus the “Favourites first” sort results. Per-user data — server must scope reads and writes to the session user.
Tag creationNew tags are only created via the recipe form, so they inherit editor auth. Filtering by existing tags is public.
Account menu & settingsProfile menu, settings page (name, email, password change), delete account. Password change and account deletion should re-verify the current password server-side.
Public without authBrowse, search, sort (except favourites), tag filtering, reading recipes and notes, log in, password reset (reset must be token-gated via email in production — the prototype applies it immediately).

Illustration & asset rules

  • logo.svg — full header lockup (chef-boy icon + wordmark, yellow), replaces the typeset wordmark; logo-no-icon.svg — wordmark alone (blue), login/reset cards and docs; star.svg — favourite star; chef.svg — login and empty library; vieja.svg — the “no recipes match” empty state; cookware.svg — missing photos; little-guy.svg — favicon. All are true vector re-exports (paths, no embedded bitmaps); chef/cookware are recoloured to ink rgba(41,39,40,0.85).
  • Favicon mark — render on a plain or brand-yellow background at 32px and 512px.
  • Recipe photos get the scanned-print treatment: contrast(1.18) saturate(0.92) brightness(0.98) plus a screen-blended overlay of photo-grunge.jpg tinted rgba(11,122,249,0.22) at 60% opacity.
  • Page graph paper shows at ~15% strength (0.85 alpha wash over the tile).
  • Max one illustration per viewport; never behind dense text; always aria-hidden when decorative.
  • Textures (paper, graph, cardboard) only on page background, header, cards and chips — never behind body copy at full strength.
  • Rotation only on illustrations, the wordmark and chips, ≤2°; never on text blocks.
  • Utility glyphs (⌕ ▾ × ↑ ↓) inherit currentColor; motion is fades ≤200ms and respects reduced-motion.
